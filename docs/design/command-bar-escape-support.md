# 命令直通车 — 转义字符支持设计方案

> 给命令直通车 (cmdbar) 的短语补全转义字符支持, 使纯字面短语也能表达
> 换行 (`\n`)、制表 (`\t`)、回车 (`\r`) 与字面反斜杠 (`\\`), 从而支持
> 多行文本短语。

## 1. 背景与目标

### 1.1 现状

命令直通车已落地三种短语形态 (见 `docs/design/command-bar-design.md`):

- **纯字面短语** — yaml 里直接写文本, 解析为 `ast.LiteralPhrase`
- **模板短语** — 含 `{expr}` 内插, 解析为 `ast.TemplatePhrase`
- **`$CC` / `$SS` 命令短语** — 表达式语言

转义序列 (`\n \r \t \\` 等) 当前**只在两条路径上生效**:

| 路径 | 处理位置 | 未知 `\X` 行为 |
|---|---|---|
| `$CC` / `$SS` 字符串字面量 | `parser/lexer.go::scanString` | 报错 `unknown escape` |
| 模板短语 | `parser/parser.go::parseTemplatePhrase` | 报错 `unknown escape` |
| **纯字面短语** | **无处理** | — (原样, `\n` 上屏为两个字符) |

纯字面短语是最朴素、最常用的短语形态。用户想配一个多行短语
(如多行签名、多行地址), 在 yaml 写 `text: "第一行\n第二行"`,
`\n` 会原样上屏成反斜杠 + 字母, 而非换行。

### 1.2 关键架构事实

不含 `$` 的纯字面短语**根本不经过 `cmdbar.Parse()`**:
`internal/dict/value_expand.go::HasExpandable` 在 value 不含 `$` 时直接
返回 `false`, `ValueExpander.Expand` 原样返回 value。因此转义解码
**不能**放在 cmdbar parser 的 `LiteralPhrase` 分支 —— 那条分支对
最常见的多行短语 (无 `$`) 永远不会被触达。解码落点必须在 dict 短语层。

### 1.3 设计目标

1. 纯字面短语支持 `\n \r \t \\` 转义, 从而支持多行文本
2. 三条路径 (`$CC` 字符串 / 模板短语 / 纯字面短语) 转义行为**完全一致**
3. 不破坏现存短语 (尤其含 Windows 路径单反斜杠的短语)
4. 转义集合保持小而稳, 为未来扩展留干净的演进空间

## 2. 转义策略

### 2.1 已决策

| 决策点 | 结论 |
|---|---|
| 「回车」语义 | **插入换行字符** (不模拟 Enter 按键注入) |
| 换行符形式 | **原样输出 `\n`** (不规范化为 `\r\n`) |
| 转义集合 | `\n` `\r` `\t` `\\` 四个 (宽松白名单) |
| 未知 `\X` | **原样保留** (反斜杠连同后续字符), 不报错 |
| 三条路径一致性 | **统一宽松** —— `$CC` 字符串 / 模板短语的未知转义报错也一并改为原样保留 |

### 2.2 策略选型: 宽松白名单 + 命名空间预留

候选策略对比:

| 策略 | 已知转义 | 未知 `\X` | 字面反斜杠 | 现存短语 | 扩展性 |
|---|---|---|---|---|---|
| 宽松白名单 (选用) | `\n\r\t\\` | 原样保留 `\X` | 单个 `\` | 不破坏 | 见下 |
| 严格白名单 | `\n\r\t\\` | 报错 | 单个 `\` | 破坏 (路径降级) | 好 |
| 通用转义 | `\n\r\t\\`… | 丢 `\` 留 `X` | 必须 `\\` | **全部破坏** | 最好 |

选用**宽松白名单**, 理由:

- 通用转义会破坏**所有**含单反斜杠的现存短语 (`C:\Users` → `C:Usersname`),
  迁移代价现在就要付; 换来的「未来可随意加单字母转义」对一个输入法短语
  语言几乎用不到。
- 宽松白名单对扩展性的损失是**有限且可控**的: 损失大小取决于未来新转义的
  匹配形状。
  - 单字母转义 (`\e` `\b`) 形状最宽, 会撞到 `C:\export` 类路径 —— 但
    输入法短语语言基本不需要这类转义。
  - 形状特定的转义 (`\uHHHH` 要求恰好 4 位 hex, `\xHH` 要求 2 位 hex)
    几乎撞不到现实短语 (`\users` 的 `sers` 非 hex, 不匹配 `\uHHHH`),
    将来随时可安全加入。
- 真正需要「插入任意字符 / PUA 码位」时, 走 `$CC(...)` 函数比扩 `\` 转义
  更干净 —— `$` 命令符空间是另一条独立的扩展通道。

### 2.3 命名空间预留 (零成本未来防护)

文档中声明: **`\` + ASCII 字母是保留命名空间**。用户不应依赖
`\` + 字母组合保持字面值; 路径中的反斜杠应写 `\\`。

这是纯文档约定, 不改变运行时行为, 但使将来若需新增转义时, 该变更从
「硬破坏」降级为「用户已被预先告知」。形状特定的转义 (`\uHHHH` 等)
本就安全, 不受此约定限制。

## 3. 实现设计

### 3.1 统一转义解码器

`internal/cmdbar` 包新增导出函数:

```go
// DecodeEscapes 宽松解码短语文本中的转义序列。
//   \n → LF, \r → CR, \t → TAB, \\ → \
// 未知 \X (含路径反斜杠) 连同反斜杠原样保留。
// 末尾孤立的 \ 原样保留。
func DecodeEscapes(s string) string
```

这是三条路径共用的**单一实现**, 杜绝行为分叉。无 `\` 时应快路径
原样返回 (`strings.IndexByte(s, '\\') < 0`)。

### 3.2 三条路径接入

| 路径 | 改动 |
|---|---|
| `$CC` / `$SS` 字符串字面量 | `lexer.go::scanString` 的 `default` 分支: 由 `return error` 改为 `lit = append(lit, '\\', next)` |
| 模板短语 | `parser.go::parseTemplatePhrase` 的 `default` 分支: 同上 |
| 纯字面短语 | dict 短语层接入 `DecodeEscapes`, 见 §3.3 |

`scanString` / `parseTemplatePhrase` 现有的 `\n \r \t \\ \" \' \{ \} \( \)`
分支保持不变; 仅未知转义分支由报错改为原样保留, 与 `DecodeEscapes`
语义对齐 (注: lexer 中 `\" \{` 等是字符串语法所需, `DecodeEscapes`
作用于已剥离引号的字面短语文本, 集合可以更小, 只需 `\n\r\t\\`)。

### 3.3 纯字面短语解码落点 (dict 层)

`internal/dict` 引入私有 helper `decodePhraseEscapes(text string) string`:
含 marker (`HasAAMarker || HasSSMarker || HasCmdbarMarker`) 的文本原样
返回 (marker 内字符串字面量的转义由各自 parser 处理); 其余调用
`cmdbar.DecodeEscapes`。这个自守卫的 helper 应用到 **PhraseLayer 的
所有候选构造点**, 因为短语候选有多条出口路径, 任何一处遗漏都会让用户
在某些上屏场景看到字面 `\n`。

落点清单 (`internal/dict/phrase.go`, 仅 `Text` 字段套 helper;
`PhraseTemplate` / `ID` 仍用原始 `e.Text` / `displayName` 以保证模板
稳定性与 ID 不漂移):

| 路径 | 触发场景 | 出口函数 |
|---|---|---|
| 静态短语全码 | 输入 = 短语 code | `Search` |
| 静态短语前缀 | 输入是 code 前缀, 空格上屏 | `SearchPrefix` (路径 2) |
| 动态短语全码 (非命令) | `$Y` 等模板变量, 全码命中 | `expandDynamicEntry` (`!res.IsCommand` 时) |
| `$AA` 字符组成员展开 | 全码命中 `$AA` group | `SearchCommand` (情况 2) |
| 字符组前缀 nav | `$AA`/`$SS` group 前缀命中 | `SearchCommand` (情况 3) + `SearchPrefix` (路径 1) |

不解码的情况:
- `$CC` 命令短语的 `Text` 来自 cmdbar `eval` (display 已由 `scanString`
  解码), **不重复解码** —— `expandDynamicEntry` 用 `!res.IsCommand`
  守卫。
- `$SS` 命令短语内嵌的 string-literal 元素由 cmdbar `scanString` 在
  解析期解码, dict 层不需要额外处理。

注意: 候选构造点散落多处是历史现状。任何新增/重构涉及候选 `Text`
字段赋值时, 必须确认对纯字面分支已套 `decodePhraseEscapes`, 否则会
重现「全码正确、前缀错误」一类的不对称 bug。

### 3.4 候选框单行渲染 (渲染层防御性处理)

候选窗口的候选标签是**单行渲染**。任何候选 `Text` 含换行符 (`\r` / `\n`)
时, 单行控件会真的换行, 导致内容溢出窗口或与相邻候选重叠。

因此在**候选渲染层**做防御性处理: 把候选显示标签里的 `\r` / `\n`
替换为一个**可见特殊符号**, 既保持单行, 又对用户有明确指示作用。

要点:

- **作用范围是渲染层, 与转义来源无关**: 不论换行符是转义解码 (`\n`)
  来的, 还是用户 / 词库里本就存在的裸换行, 一律在渲染时替换。这是候选
  UI 对控制字符的统一防御, 不是转义特性的局部逻辑。
- **只改显示标签, 不改候选 `Text`**: 候选 `Text` (真正上屏的内容)
  保留真实换行符不变, 上屏多行文本仍然多行。
- 替换符号默认用 `↵` (U+21B5), 作为可调常量集中定义 (与
  `ui.CmdbarCandidatePrefix` 同类, 便于按字体兼容性调整)。`\r\n`
  连续出现折叠为单个符号, 避免出现两个。
- 落点在候选文本送入单行控件渲染之前的统一钩子处 (实现计划阶段
  定位精确函数); 应覆盖所有候选来源 (短语 / 码表 / 用户词库 / 命令)。

### 3.5 设置 UI 往返

wind_setting 短语编辑器中, **存储与编辑展示统一用原始转义形态**
(`第一行\n第二行` 字面)。编辑器不做解码, 也不做编码, 因此不存在
「编辑一次被改坏」的往返风险。该写法与 `$CC` 字符串字面量一致。

## 4. 测试要求

- `cd wind_input && go test ./internal/cmdbar/...` 全绿。
- `DecodeEscapes` 表驱动单测: `\n\r\t\\` 解码、未知 `\X` 保留、
  末尾孤立 `\`、无 `\` 快路径、连续转义、Windows 路径不被破坏。
- `scanString` / `parseTemplatePhrase` 未知转义不再报错的回归用例。
- dict 层: 多行纯字面短语候选 `Text` 含真实换行的端到端用例。
- 候选渲染层: `\r` / `\n` / `\r\n` 替换为单个 `↵` 符号、候选 `Text`
  本身不被修改、非换行候选不受影响。
